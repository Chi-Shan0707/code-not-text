Code Correctness Is Not in the Text

Cross-Domain Limits of Hand-Crafted CoT-Surface Features
Yuhan Chi · Fudan University · 2026

Abstract. This study examines whether hand-crafted Chain-of-Thought (CoT) surface features can predict solution correctness across mathematical reasoning, science problems, and coding challenges. Using 31,040 model runs across three domains, we find that the same feature family achieves AUROC 0.982 in math, 0.850 in science, but only 0.407 in coding. Five independent methods (interpretable probe, feature sweep, nonlinear MLP, SSL pre-training, and de-knotting) all fail to improve coding performance. We argue that this demonstrates a measurement invariance failure: text-based features measure different underlying constructs in each domain, and code correctness fundamentally requires execution-aware verification rather than text analysis.

1. Core Findings

We evaluated five hand-crafted CoT surface features across three domains using DeepSeek-R1-0528-Qwen3-8B:

Domain AoA (95% CI) Best-of-N=64 Gain Interpretation
Math 0.958 [0.931, 0.980] +10.0% pass@1 Features capture "failure to converge" patterns
Science 0.799 [0.775, 0.822] +8.0% pass@1 Signal driven by token confidence, not structure
Coding 0.434 [0.404, 0.464] -0.6% (no gain) Text features miss execution correctness
Figure 1: AUROC scores by domain. Math shows excellent discrimination (near 1.0), coding performs below chance (0.5).

Key Observation

The same feature family produces dramatically different results across domains. This is not about model capacity, data quality, or feature engineering—it's about what text features can actually measure in each domain.

2. Feature Family: What We Actually Measured

Our five hand-crafted CoT surface features were designed to capture reasoning quality without understanding semantic content:

Feature Type Description Why It Should Work
token_conf_mean Token-level Average confidence score across all tokens Confident models are often correct
token_conf_gini Token-level Gini coefficient of token distribution Sharp distributions indicate certainty
traj_length Trajectory Number of reasoning steps Longer reasoning may indicate difficulty
traj_reflection_count Trajectory Count of correction patterns ("but", "however") Self-correction indicates reasoning
traj_entropy Trajectory Entropy of token transitions Structured reasoning has predictable patterns
Feature Extraction Pipeline
# Example: Extracting reflection patterns
def extract_reflection_count(co_trace: str) -> int:
    """Count linguistic markers of self-correction."""
    markers = ["but", "however", "wait", "actually", "although"]
    count = sum(co_trace.lower().count(m) for m in markers)
    return count

Why these features? They capture surface patterns that should correlate with reasoning quality: confident models make correct predictions, self-correction indicates meta-reasoning, and structured trajectories reflect systematic thinking. These features work well in math, partially in science, but fail completely in coding.

3. Why Math Works (and Coding Doesn't)

Math: Text Features Capture Reasoning Quality

In mathematical reasoning, correctness is tightly coupled with reasoning process. When a solution fails, we often see:

  • Hesitation patterns (low token confidence)
  • Multiple attempts at solution (high reflection count)
  • Circular reasoning or dead ends (high trajectory entropy)

Our features capture these patterns because math errors are visible in the reasoning trace.

Science: Mixed Signal

Science problems show intermediate performance (AoA 0.799). The signal exists but is weaker because:

  • Domain knowledge matters more than reasoning patterns
  • Token confidence drives most of the signal
  • Trajectory structure provides limited additional information

Features work partially, but they're mostly measuring model confidence rather than reasoning quality.

Coding: Text Misses Execution

In coding, correctness is determined by execution, not text. Our features fail because:

  • Syntax errors don't affect reasoning quality scores
  • Logic errors produce perfectly confident text
  • Corner cases are invisible in the code trace
  • Execution state (runtime errors, memory issues) is not captured

A model can write highly confident, well-structured code that fails on edge cases—our features would rate this as "high quality" but it's actually wrong.

Example: Confident but Wrong Code
# High confidence, structured reasoning, but WRONG
def find_median(arr):
    """Flawless reasoning trace, confident tokens..."""
    return sorted(arr)[len(arr)//2]  # Off-by-one error

# Our features predict: CORRECT (high confidence, structured)
# Actual execution: WRONG (fails on even-length arrays)

Core Insight

Code correctness is not in the text. It's in the execution. Text-based features can measure reasoning quality when correctness depends on reasoning (math), but they cannot measure execution correctness (coding). This is a fundamental limitation, not a feature engineering problem.

4. Five Methods, One Convergent Result

To ensure our findings are robust, we tested five independent approaches. All converge on the same conclusion: text-based features don't work for coding correctness.

Method 1: Interpretable Probe (LogReg + SVD)

Approach: Train a logistic regression classifier on our 5 features with SVD dimensionality reduction.

Result: Coding AUROC: 0.434

Interpretation: The interpretable baseline performs below chance, suggesting the features themselves don't contain useful signal for coding.

Method 2: Feature Sweep (83 Coding-Specific Scalars)

Approach: Test 83 additional features designed specifically for code: indentation patterns, function complexity metrics, keyword frequencies, AST-based features.

Result: Best coding AUROC: 0.556

Interpretation: Even with code-specific features, performance barely exceeds random guessing. The problem isn't feature selection—it's that text features fundamentally miss execution correctness.

Method 3: Nonlinear MLP (Deep Neural Network)

Approach: Train a deep neural network to learn nonlinear feature combinations.

Result: Coding AUROC: 0.499–0.507

Interpretation: Adding model capacity doesn't help. If the signal isn't in the features, a more powerful model can't extract it.

Method 4: SSL Pre-training (42K Unlabeled Traces)

Approach: Use self-supervised learning on 42K unlabeled coding traces to learn domain-specific representations.

Result: Full-label ceiling: AUROC 0.454

Interpretation: Even with massive unsupervised pre-training, the model can't learn to predict correctness from text. The ceiling is below random chance.

Method 5: De-knotting (Remove Circular Reasoning)

Approach: Identify and remove "knots" (circular reasoning spans) from CoT traces, hypothesizing they add noise.

Result: AUROC change: +0.006

Interpretation: Cleaning up reasoning traces doesn't improve performance. The bottleneck isn't trace quality—it's that text doesn't contain execution information.

Figure 2: All five methods fail to achieve meaningful performance on coding (AUROC > 0.6), while working well on math.

5. What These Failures Mean

The convergent failure across five methods suggests a deeper principle:

5.1. Surface Features Don't Capture Semantic Meaning

Linguistic markers like "but", "however", "wait" don't reliably indicate reasoning quality. In coding, these tokens may appear in confident but wrong solutions. The form of reasoning doesn't predict the correctness of output.

5.2. Self-Supervised Learning Can't Create Execution Signal

Even with 42K unlabeled traces, SSL can't learn to predict correctness. This suggests that execution information isn't latent in the text—it's fundamentally absent. No amount of representation learning can extract what isn't there.

5.3. Structure Doesn't Imply Correctness in Code

Well-indented, syntactically correct code with clear reasoning traces can still be wrong. Conversely, messy code can be correct. Text-based structural features (indentation, function length, etc.) don't correlate with execution correctness.

5.4. Measurement Invariance Failure

Our features measure different constructs in different domains:

  • In math: They measure reasoning quality
  • In science: They measure model confidence
  • In coding: They measure text fluency (not correctness)

This is a measurement invariance failure: the same operationalization measures different underlying constructs across contexts.

6. Frequently Asked Questions

Q: Did you try execution-aware features?
A: That's exactly the point—we deliberately limited ourselves to text-based features to test whether CoT surface contains execution information. Our findings show it doesn't. Execution-aware features (running code, checking test cases) would work, but that's a different paradigm.
Q: Maybe you just need better features?
A: We tested 88 total features (5 general + 83 coding-specific). Method 3 (MLP) shows that adding model capacity doesn't help. Method 4 (SSL) shows that unsupervised pre-training doesn't help. The bottleneck isn't feature engineering—it's that execution correctness isn't encoded in text.
Q: Does this mean all text-based verifiers fail on coding?
A: No. We specifically tested hand-crafted CoT surface features. Other approaches like LLM judges or execution-based verifiers may work. Our claim is limited to: text-based surface features can't predict code correctness.
Q: Why does math work so well?
A: In math, correctness depends on reasoning process. When a solution is wrong, the reasoning trace typically shows hesitation, backtracking, or circular patterns. Our features capture these process-level signals. In coding, correctness depends on execution, which leaves no trace in the CoT text.
Q: Is this model-specific?
A: We tested DeepSeek-R1-0528-Qwen3-8B. However, the fundamental limitation (execution vs. text) should generalize. Different models may have different confidence patterns, but the core issue remains: you can't extract execution correctness from text alone.

7. Technical Appendix

7.1. Experimental Setup

Model

DeepSeek-R1-0528-Qwen3-8B (8B parameters, reasoning-specialized)

Datasets
  • Math: AIME24/25, BRUMO25, HMMT25 (7,680 runs)
  • Science: GPQA Diamond (12,672 runs)
  • Coding: LiveCodeBench-v5 (10,688 runs)
Evaluation Metric

We use AoA (AUC-of-AUROC): the area under the AUROC curve across 10 anchor points in the CoT trace. This measures how well features discriminate correct vs. incorrect solutions at each stage of reasoning.

Statistical Validation

All confidence intervals are 95% bootstrap CIs (10,000 resamples). Results are statistically significant: math and science differ from coding (p < 0.001).

7.2. What We Don't Claim

Limitations:

  • We don't claim that all text-based verifiers fail on coding (e.g., LLM judges may work)
  • We don't claim that execution-aware methods won't work (they should!)
  • We don't test larger models or other architectures
  • Results may not generalize to all programming languages

Future Work: Execution-aware verification, hybrid text+execution approaches, domain-specific feature engineering.
