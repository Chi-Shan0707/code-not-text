Code Correctness Is Not in the Text

Why Chain-of-Thought Surface Features Work for Math but Fail for Coding
Yuhan Chi · Fudan University · 2026

Abstract. Hand-crafted Chain-of-Thought (CoT) surface features achieve AUROC 0.982 in mathematical reasoning but only 0.407 in coding. Why? This study argues that math reasoning is fundamentally linear—"predict next token" captures the essence of step-by-step problem solving. But coding involves non-linear control flow: loops, branches, and state management that natural language cannot faithfully represent. Five independent methods (feature sweep, SSL pre-training, nonlinear MLP, de-knotting, and interpretable probes) all fail to improve coding performance, demonstrating that execution correctness is not in the text—it's in the runtime.

1. The Core Motivation: Why Math and Coding Are Different

Before presenting results, we need to understand why we expected text-based features to work in the first place, and why they fundamentally fail for coding.

📐 Math & Science: Linear Reasoning

Mathematical and scientific reasoning is inherently sequential:

  • Solve step A → verify → proceed to step B
  • "Predict next token" aligns with "predict next reasoning step"
  • Mistakes create visible patterns: hesitation, backtracking, circular reasoning
  • Text features capture reasoning quality

Result: AUROC 0.982 (math), 0.850 (science)

💻 Coding: Branching & State

Programming involves control flow that natural language cannot represent:

  • Loops: Iterative state updates that linear text flattens
  • Branches: Conditional logic with multiple paths
  • State: Variable bindings, memory, runtime errors
  • Confident text ≠ correct code (off-by-one errors, edge cases)

Result: AUROC 0.407 (below random!)

The "Knot" Hypothesis

We call this the knot problem: natural language (a linear medium) cannot faithfully represent non-linear control flow without "knots"—places where state is lost, branches get tangled, and the text becomes unreadable. In math, reasoning naturally follows a linear path. In coding, the mental model involves branches and loops that get flattened into linear text, creating a representational mismatch.

What These Examples Show

These aren't "reasoning errors" in the traditional sense. The model isn't making logical mistakes—it's drowning in control flow. The text shows:

  • High confidence (token_conf_mean is normal)
  • High "reversal density" (lots of "but", "however", "wait")
  • But the reversals aren't about self-correction—they're about tracking branching state

Key insight: In math, "but wait" means "I found a mistake." In coding, "but wait" means "I lost track of which branch I'm in."

# Example: Computing reflection features from CoT trace
def compute_reflection_density(co_trace: str) -> float:
    """Count linguistic markers of self-correction per 1K chars."""
    reflection_markers = ["but", "however", "wait", "actually", "although"]
    total_count = sum(co_trace.lower().count(m) for m in reflection_markers)
    return (total_count / len(co_trace)) * 1000

5. Five Methods, One Convergent Result

We tested five independent approaches to ensure our findings are robust. All converge on the same conclusion: text-based features can't predict code correctness.

Method 1: Feature Sweep (83 Coding-Specific Features)

Approach: Designed 83 features specifically for code: indentation depth, function complexity, keyword frequencies, AST-based metrics, loop nesting depth.

Result: Best coding AUROC: 0.556

Interpretation: Even with code-specific features, performance barely exceeds random. The problem isn't feature selection—it's that execution correctness isn't in the text.

Method 2: SSL Pre-training (42K Unlabeled Traces)

Approach: Used self-supervised learning (like BERT) on 42K unlabeled coding traces to learn domain-specific text structure.

Result: Full-label ceiling: AUROC 0.454

Interpretation: SSL learns text structure (syntax patterns, common phrases), but code correctness isn't a text-structure problem—it's an execution problem. No amount of representation learning can extract what isn't there.

Method 3: Nonlinear MLP (Deep Neural Network)

Approach: Trained a deep neural network to learn nonlinear feature combinations.

Result: Coding AUROC: 0.499–0.507

Interpretation: Adding model capacity doesn't help. If the signal isn't in the features, a more powerful model can't create it.

Method 4: De-knotting (Remove Circular Reasoning)

Approach: Identified and removed "knots" (circular reasoning spans) from CoT traces, hypothesizing they add noise.

Result: AUROC change: +0.006

Interpretation: Cleaning up reasoning traces doesn't help. The "knots" in coding aren't reasoning errors—they're structural. Removing them doesn't fix the fundamental mismatch.

Method 5: Interpretable Probe (LogReg + SVD)

Approach: Logistic regression with SVD dimensionality reduction on our 5 core features.

Result: Coding AUROC: 0.434

Interpretation: The interpretable baseline performs below chance, confirming that the features themselves don't contain useful signal for coding.

Figure 2: All five methods fail on coding (AUROC < 0.6), while working well on math.

6. What These Failures Mean

6.1. SSL Learns Structure, Not Correctness

Self-supervised learning (Method 2) successfully learns text structure—just like BERT learns sentence structure. But code correctness isn't about text structure. It's about:

  • Runtime behavior (does this loop terminate?)
  • Edge cases (what if the array is empty?)
  • State correctness (did I update the right variable?)

These aren't in the text, so SSL can't learn them.

6.2. Feature Engineering Can't Create What Isn't There

Method 1 tested 88 total features (5 general + 83 coding-specific). Method 3 added non-linear modeling. Method 4 cleaned up the text. None helped. This converges on one conclusion: execution correctness is not recoverable from CoT text.

6.3. The "Knot" Problem Is Structural, Not Superficial

The heavy branch examples show that coding CoT traces get "knotted"—not because of bad reasoning, but because the medium (linear text) cannot faithfully represent the message (branching control flow). De-knotting (Method 4) fails because you can't unknot a knot by removing words—the knot is in the structure, not the surface.

6.4. Measurement Invariance Failure

Our features measure different constructs in different domains:

  • In math: They measure reasoning quality (does the trace show "failure to converge" patterns?)
  • In coding: They measure text fluency (is the CoT coherent?), not correctness

This is a measurement invariance failure: the same operationalization measures different underlying constructs across contexts.

7. Frequently Asked Questions

Q: Did you try execution-aware features?
A: That's exactly what we're not testing. We deliberately limited ourselves to text-based features to test whether CoT contains execution information. Our findings show it doesn't. Execution-aware features (running code, unit tests) would work, but that's a different paradigm—we're testing the limits of text-only verification.
Q: Why does SSL fail if BERT works so well on text?
A: SSL (like BERT) learns text structure. BERT can predict the next word in a sentence because sentences have structure. But code correctness isn't about predicting the next word in the CoT trace—it's about predicting runtime behavior. SSL can learn that "def function_name(" usually follows "return ", but it can't learn whether the function is correct.
Q: Is this model-specific?
A: We tested DeepSeek-R1-0528-Qwen3-8B. However, the fundamental limitation (linear text vs branching execution) should generalize. Different models may have different confidence patterns or writing styles, but the core issue remains: you can't extract execution correctness from text alone.
Q: What about LLM judges or GPT-4 verification?
A: Those are different approaches. We're testing hand-crafted surface features. LLM judges use semantic understanding (they read the code and "think" about it). Our question was: can we predict correctness from surface patterns (confidence, trajectory shape) without understanding? Answer: no for coding, yes for math.
Q: Does this mean text-based verifiers are useless for coding?
A: No. It means hand-crafted CoT surface features don't work. Other approaches like LLM judges, execution-based verifiers, or hybrid methods may work. Our claim is limited to: surface features extracted from CoT text cannot predict code correctness.

8. Technical Appendix

8.1. Experimental Setup

Model

DeepSeek-R1-0528-Qwen3-8B (8B parameters, reasoning-specialized)

Datasets
  • Math: AIME24/25, BRUMO25, HMMT25 (7,680 runs)
  • Science: GPQA Diamond (12,672 runs)
  • Coding: LiveCodeBench-v5 (10,688 runs)
Evaluation Metric

We use AoA (AUC-of-AUROC): the area under the AUROC curve across 10 anchor points (10%, 20%, ..., 100%) in the CoT trace. This measures how well features discriminate correct vs. incorrect solutions as the reasoning unfolds.

Statistical Validation

All confidence intervals are 95% bootstrap CIs (10,000 resamples). The math-coding gap is statistically significant (p < 0.001).

Annotation Pipeline

Coding domain was annotated using three complementary methods: GLM-4 (large language model), Python scripts (pattern-based), and human annotation. This multi-method approach ensured comprehensive coverage of "knot" patterns.

8.2. What We Don't Claim

Limitations:

  • We don't claim that all text-based verifiers fail on coding
  • We don't claim that execution-aware methods won't work (they should!)
  • We don't test larger models or other architectures
  • Results may not generalize to all programming languages

Future Work: Execution-aware verification, hybrid text+execution approaches, domain-specific feature engineering for other programming tasks.
